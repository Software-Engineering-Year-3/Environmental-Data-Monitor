Introduction

The Environmental Data Monitor is a cross-platform mobile application, developed with C# MAUI, designed to assist a regional environmental agency in the surveillance of air quality, water quality, and meteorological conditions. The application offers real-time sensor status on an interactive map, historical data visualisation and analysis, alerts for threshold violations, and user-role administration. It employs a simulated data methodology for environmental measurements (via in-memory services and integrated CSV resources) and a mock user store persisted in app preferences, allowing a completely operational front-end without a fully functional backend.

Detailed Implementations & Data-Loading Journey

Database & SQLite Attempts

I attempted to use EF Core with SQLite:

  • Packages: Microsoft.EntityFrameworkCore.Sqlite, Design
  • Created AppDbContext with multiple DbSets
  • Generated migrations

Obstacles: target-framework mismatch (net8.0-android vs netstandard2.1), missing sqlite3.dll, runtime binding errors.

CSV File Loading

Placed .csv files under Resources/Raw, used CsvHelper. Encountered ANR (jank) on large files and inconsistent formatting, despite try/catch safeguards.

Final Mock-First Approach

  • SensorService.LoadMockSensors(): 13 sample sensors (air, water, weather).
  • ViewModels attempt CSV load, then fallback to static mock arrays with simulated delays.

This guaranteed consistent functionality and allowed focus on UX and user stories.

Unit Test Efforts

Created an NUnit test project targeting .NET 8.0 and 6.0:

  • SensorServiceTests.LoadMockSensors_ShouldPopulateList()
  • SensorDashboardViewModelTests.LoadSensorsCommand_ShouldPopulateObservableCollection()

Tests could not run on MAUI runtime (net8.0-android) due to emulator/framework constraints, but have been preserved for future integration.

Issue 1: Locate & Navigate to Sensors

Summary: As an Environmental Scientist, I needed to locate and navigate to field sensors.

Implementation: Developed SensorMapPage with map pins labeled with Name, Type, and Status. Pins sourced from SensorService.LoadMockSensors(). Navigation triggered via MapCommand in SensorDashboardViewModel, passing query parameters for single-sensor focus.

The page’s OnAppearing override populates pins, wraps AddPin in try/catch, zooms to a 2 km radius, and alerts on errors. Manual testing on USB-connected Android and emulator confirmed pinch-to-zoom, centring, and back navigation.

Future: integrate device GPS or coordinate transmission.

Issue 2: Report Sensor Malfunctions

Summary: Technicians must swiftly document sensor anomalies.

I created ReportIssuePage with:

  • Read-only sensor identifier (populated via navigation parameters)
  • Editor for issue description
  • Optional Entry for reporter name
  • Submit Report button

The OnSubmitClicked handler validates inputs (stub), shows a “Success” alert, and navigates back via Shell.Current.GoToAsync(".."). Manual testing confirmed responsiveness and correct navigation on both device and emulator. Persistent storage is deferred for future enhancements (e.g., local DB or remote API).

Issue 3: Manage User Access & Roles

Summary: Administrators must oversee user accounts and roles.

I implemented UserManagementPage backed by MockUserStore and UserSession:

  • CollectionView of users (Name, Email, Role)
  • Picker for role changes with confirmation dialog
  • Delete button with confirmation
  • Persistence via JSON in Preferences with try/catch fallback
  • UserSession controls visibility of admin features

Manual testing ensured session persistence, correct UI updates, and safe confirmation flows.
